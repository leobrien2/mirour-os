🔴 Core Problem — Tagging is Everything
The problem: Product data from brands is messy. CSVs have no images, no tags, minimal descriptions. Web scraping worked for Soberish demo but isn't scalable. Every brand's data is different and incomplete.

The solution (from team meeting): LLM-powered auto-tagging. Onboarding form captures brand context (who they are, what they sell). Batch 100 products → send to LLM with context → LLM returns consistent tags → store in DB. Prompt engineering ensures tag consistency across batches.

Why this is IP: "If we can figure out a way for this software to auto-tag and tag in the moment, we will win." — Kelley. Two-way sync between Mirour tags and ecom platforms (Shopify, etc.) means brands' agentic readiness improves just by using Mirour. Tags flow back to their stores. Nobody else does this.
Priority: Build auto-tagging system. Test with Glad & Young. This unblocks everything.

"Dresses with pockets" → AI finds all products with "dress" + "pocket" in description → adds tag
Tags compound over time — system gets smarter with every flow built
Complex but doable — Tanmay confirmed. Prompt engineering + proper linking.

Phase 3: Two-Way Ecom Sync (FUTURE — The Moat)

Mirour talks to Shopify bidirectionally. Tags that Mirour creates flow back to the brand's store. Their products become searchable + agentic-ready. Brands learn from how people search in flows → feed that intelligence to ecommerce. "The heavy lift happens with Mirour, tested and proven in-store, then feeds back to their ecom platform." — Kelley

WORKING

Flow Builder

Node graph editor with 7 step types. Supports quiz questions, conditional logic, tag assignment, product recommendations. Side-by-side preview shipped Mar 20.

⚠ Complex for non-engineers. From meeting: hide recommendation strategy logic (any/all/zone), show tags instead. Add tag-based product display mode + tags on buttons.

WORKING

Flow Runner

Customer-facing quiz at /f/[formId]. Renders all step types, calculates tags from answers, filters products by tag match. 3,000+ line file doing 3 jobs (UI + recs + data capture).

✓ Core engine is solid. Returns right products when tags exist.

WORKING

Tag-Based Recommendations

3 strategies: "any" match, "all" match, "zone-first". Quiz answers accumulate tags → filterProducts() finds matches. Works perfectly when product_tags table is populated.

⚠ From meeting: keep standard (any/all), hide logic from users. Add new mode: tag-based display — just "show everything with this tag" as simple filter. Plus tag on buttons for direct routing.

NOT WORKING

Product Import Pipeline

No automated import exists. Brands export CSV from POS → someone manually processes it → products sit in CSV never imported to Supabase. Store integrations UI saves API keys but NO code calls those APIs.

🔴 This is THE blocker. Every pilot is stuck here.

GHOST TABLE

Analytics Events

analytics_events table created in migration with full schema (event_type, payload, location_id). Zero code writes to it. The /analytics page has nothing to show because no events are tracked.

🔴 Can't measure anything. No funnel, no conversion, no drop-off.

GHOST TABLE

Customer Tags

customer_tags table exists. lib/tags.ts has appendTags() and getSessionTags() functions ready to go. But the flow runner NEVER calls them. Customer profiling data vanishes after each session.

🔴 One function call to fix. Huge value unlock.

GHOST TABLE

Form Visits / Flow Sessions

form_visits tracks QR scans/page visits. flow_sessions tracks start/complete. Both tables exist, neither written to. Can't measure engagement or completion rates.

PARTIALLY BUILT

Store Integrations

IntegrationsModal lets users save Lightspeed, Shopify, Squarespace API keys. Keys are stored in store_integrations table. But NO code ever calls those APIs to sync products. UI theater.

PARTIALLY BUILT

Profile Engine (M3)

Weighted scoring from profile_rules, hard branch for non-drinker, tiebreaker logic. Called by flow runner but has hardcoded "wellness-seeker" ID that won't match real data. Rules not populated for any pilot.

⚠ Premature. Tags alone do the job for now.

OVERBUILT

Zone System

Maps products to physical store areas. zone_tags junction table exists. "zone-first" recommendation strategy built. No pilot uses zones. Adds cognitive overhead to flow builder for zero current value.

⚠ Hide for now. Bring back when a pilot asks for wayfinding.

WORKING

Customer Info Capture

Captures phone/email during flows. Customer lookup via token. Creates customer records. Works, but unclear if repeat visitors are matched to previous sessions.

PARTIALLY BUILT

Inventory Management

/inventory page exists. Products can be viewed/edited. But product import is manual and tags must be assigned one by one. No bulk operations. No POS sync.

FUTURE

AI Flow Generation

Import products → AI reads categories/descriptions → auto-generates a quiz flow with tag mappings. Eliminates the blank canvas problem. The #1 simplification for onboarding.

💡 POS categories ARE the quiz. "What are you in the mood for?" writes itself from product data.

FUTURE

LLM reads product name + description + category → generates tags automatically. "Kin Euphorics High Rhode" → functional, adaptogen, no-alcohol, social, energize. Turns week-long onboarding into same-day.

FUTURE

Customer Insights Dashboard

Once analytics + customer tags work: "62% of customers want non-alcoholic" / "Top vibe: calm evening" / "THC products have 3x save rate". Makes $199/mo feel like a steal.

FUTURE

Multi-Channel Distribution

Flow runner already works as standalone page. Package as: embed widget for brand websites, shareable link for email campaigns, Instagram story link. Distribution is free, just not packaged yet.

FUTURE

Live POS Sync

Use the API keys already stored in store_integrations to auto-sync products nightly. New products auto-tagged and available in flows. Brands never touch a CSV again.

FUTURE

Flow Wizard (Simple Mode)

✂️ Simplify — What to Hide, Default, or Cut
The code is more complete than the team realizes. The problem isn't missing features — it's too much exposed at once. These changes make the build easier without rewriting anything.

Flow Creation

Now: Blank canvas → build from scratch → manually tag each answer
Better: Import products → system generates draft flow → tweak and launch

The current flow builder is a power tool. For pilot onboarding you need a wizard. Hide the node editor behind "Advanced" and lead with a 3-step setup.

Recommendation Strategies

Now: 3 strategies to choose from (any / all / zone-first)
Better: Default to "any match" — hide the choice entirely

Every pilot needs "any". The other two add decision overhead with zero current value. Bring them back as a pro feature when someone actually needs "all match."

Node Types

Now: 7 types (welcome, message, question, customer-info, recommendation, product-showcase, complete)
Better: 3 types for pilot flows (welcome, quiz question, recommendations)

Message and product-showcase are edge cases. Customer-info can be a toggle on the complete step. Fewer types = fewer things to explain, fewer things to break.

Profile Engine

Now: Weighted scoring + hard branches + archetype matching + tiebreakers
Better: Tags alone handle recommendations for all current pilots

Profile engine requires populated profile_rules (none exist) and has hardcoded IDs. Tags do the same job with no setup. Revisit when you have 50+ products per category.

Zone System

Now: Products mapped to physical zones, zone_tags table, zone-first filtering
Better: Hide zones entirely. Just tags + products.

No pilot uses zones. It adds a layer of mapping that makes the flow builder harder to understand. Bring it back when a pilot asks "can I guide people to aisle 3?"

Onboarding

Now: Export CSV → manually map tags → someone imports to Supabase → build flow from scratch
Better: Connect POS → auto-import → auto-tag → auto-generate flow → brand tweaks

Current onboarding is consulting disguised as a product. Each pilot takes weeks. The auto pipeline turns it into hours. This is the difference between 5 pilots and 50.
